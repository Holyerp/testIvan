# Claude Development Workflow Rules

This document defines the mandatory workflow rules for all development tasks on this project.

## 1. Internationalization (i18n)

**MANDATORY**: All features MUST include English and German translations.

- Do NOT count any task as finished until translations are implemented
- Use a translation system (e.g., react-i18next or similar)
- Translation files location: `/app/locales/`
  - `en.json` - English (default)
  - `de.json` - German
- All user-facing text must use translation keys
- Include translations for:
  - UI labels and buttons
  - Form validation messages
  - Error messages
  - Success notifications
  - Page titles and descriptions

## 7. Database Schema Changes

**MANDATORY**: Always use migrations for schema changes, never `db push`.

### Why Migrations Matter

- `db:push` only syncs locally - production/staging won't get the changes
- Migrations create trackable, deployable SQL files
- Railway (and other platforms) only run `prisma migrate deploy`

### Schema Change Workflow

When modifying `prisma/schema.prisma`:

1. **Make your schema changes** in `schema.prisma`
2. **Create a migration**: `npm run db:migrate` (will prompt for name)
3. **Review the generated SQL** in `prisma/migrations/[timestamp]_[name]/migration.sql`
4. **Test locally** - migration runs automatically with `db:migrate`
5. **Commit the migration file** along with schema changes
6. **Deploy** - Railway will run `prisma migrate deploy` automatically

### Commands

| Command                     | Use When                                               |
| --------------------------- | ------------------------------------------------------ |
| `npm run db:migrate`        | Making schema changes (creates migration + applies it) |
| `npm run db:migrate:create` | Create migration without applying (for review first)   |
| `npm run db:migrate:deploy` | Apply pending migrations (production use)              |
| `npm run db:push:force`     | Emergency only - bypasses migration requirement        |
| `npm run db:reset`          | Reset database and re-run all migrations               |

### DO NOT

- ❌ Run `npx prisma db push` directly
- ❌ Use `npm run db:push` (it's blocked with a warning)
- ❌ Modify production database manually
- ❌ Skip committing migration files

### DO

- ✅ Always use `npm run db:migrate` for schema changes
- ✅ Give migrations descriptive names (e.g., `add_user_avatar_field`)
- ✅ Review generated SQL before committing
- ✅ Test migrations locally before pushing

## 8. SOLID & DRY Principles

**MANDATORY**: All code MUST follow SOLID and DRY principles.

### SOLID Principles

| Principle                     | Rule                                                                                                                                           | Example                                                                   |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| **S** - Single Responsibility | Each module, class, or function does exactly one thing. A service handles business logic, a route handles HTTP, a component handles rendering. | `shipment.server.ts` handles shipment logic, not invoice logic            |
| **O** - Open/Closed           | Design for extension without modification. Use composition, interfaces, and dependency injection.                                              | Add new cost categories without modifying existing cost calculation logic |
| **L** - Liskov Substitution   | Subtypes must be substitutable for their base types. Error subclasses must behave as `AppError`.                                               | `NotFoundError` can be used anywhere `AppError` is expected               |
| **I** - Interface Segregation | Keep interfaces small and focused. Don't force consumers to depend on methods they don't use.                                                  | A read-only service interface doesn't include write methods               |
| **D** - Dependency Inversion  | Depend on abstractions, not concretions. Services receive their dependencies (e.g., tenant-scoped DB client) as parameters.                    | `createShipment(db, data)` receives the DB client, doesn't import it      |

### DRY (Don't Repeat Yourself)

- **Extract shared logic** into reusable service functions, utilities, or hooks when the same pattern appears 2+ times
- **Shared validation schemas** go in `app/lib/validation.ts` — reuse across routes
- **Shared UI patterns** become components — don't duplicate form layouts or table structures
- **Shared query patterns** become service functions — don't repeat the same Prisma query in multiple routes
- **Translation keys** are reused — don't create duplicate keys across namespaces

### Applying These Principles

When writing or reviewing code, verify:

- [ ] Each file/function has a single, clear responsibility
- [ ] Service functions accept dependencies as parameters (especially `db: TenantDb`)
- [ ] Repeated logic (3+ lines appearing 2+ times) is extracted into shared functions
- [ ] Validation schemas are defined once and reused
- [ ] Components are composable and don't duplicate UI patterns
- [ ] New features extend existing abstractions rather than duplicating them

## 9. Technical Debt Management

**MANDATORY**: When touching any file, fix related technical debt incrementally.

### Route File Type Fixes (TD-001)

When modifying any route file in `app/routes/`, **ALWAYS** fix the `json()` wrapper type inference issue:

1. **Remove the `json()` wrapper** from loader/action returns
2. **Return data objects directly** for proper type inference
3. **Verify** `useLoaderData<typeof loader>()` types resolve correctly

**Before (problematic):**

```typescript
import { json } from "~/lib/response";

export async function loader({ request }: Route.LoaderArgs) {
  const data = await getData();
  return json({ data }); // Breaks type inference
}
```

**After (correct):**

```typescript
export async function loader({ request }: Route.LoaderArgs) {
  const data = await getData();
  return { data }; // Direct return - types work
}
```

**Edge cases:**

- For error responses, use `throw new Response("message", { status: 404 })`
- For custom headers, use `new Response(JSON.stringify(data), { headers: {...} })`

### General Technical Debt Rules

- Check `/docs/developer/TECH_DEBT.md` for known issues
- When fixing a file, check if it has related tech debt items
- Update tech debt tracker when items are resolved
- Don't introduce new technical debt without documenting it
