# Screen Inventory — Navigation Map & API Mapping (Web CMS, Mobile)

**Version:** 1.0
**Last Updated:** 2026-05-11
**Status:** Active

**MANDATORY for projects whose UI surface area is large enough that a consolidated screen map adds value — specifically web CMS / admin panels, mobile apps (native or RN), and web apps with a non-trivial admin area. Optional for simple SPAs, API-only services, and CLI tools.**

The artifact is a single navigable index of every screen + its navigation hierarchy + which API endpoints it consumes. The API-endpoint columns are **derived from frontend stories** (per `.claude/rules/screen-driven-backlog.md`) — they are not hand-maintained, to prevent drift between the map and the backlog.

---

## 1. When This Rule Applies

| Project type | Screen inventory required? |
|--------------|----------------------------|
| Web CMS / admin panel | ✅ Required |
| Mobile app (native iOS/Android) | ✅ Required |
| Mobile app (React Native / Flutter / Expo) | ✅ Required |
| Web app with non-trivial admin area | ✅ Required (covers the admin screens at minimum) |
| Simple web SPA (≤ 5 screens) | ➖ Optional |
| API-only backend, CLI tool, library | ❌ Not applicable |

The default trigger inside `/init-project`: project type `[2] Backend + Mobile`, `[3] Backend + Web + Mobile`, or `[4] Web Only` scaffolds the artifact automatically. `[1] Backend Only` skips it.

## 2. The Artifact

**Location:** `.project-management/input/screens/screen-map.md`

For large projects (> ~40 screens) split per domain: `.project-management/input/screens/<domain>.md` (e.g. `admin.md`, `customer-app.md`, `auth.md`) + `.project-management/input/screens/README.md` master index.

**File-size budget:** ≤ 300 lines per file (templates limit per `.claude/rules/documentation.md` §2.1). If a single domain exceeds 300, split further.

**Two sections per file** (both required):

### 2.1 Navigation Hierarchy (hand-curated)

ASCII tree showing the routing / navigation structure. This is **new architectural information** that does not exist anywhere else in the project — it is what the rule primarily adds.

For **mobile** (stack/tab/drawer pattern):
```
RootStack
├── AuthStack (unauthenticated)
│   ├── LoginScreen
│   ├── SignupScreen
│   └── PasswordResetScreen
└── MainTabs (authenticated)
    ├── HomeTab
    │   ├── HomeScreen
    │   └── ProductDetailScreen
    ├── SearchTab
    └── ProfileTab
        ├── ProfileScreen
        └── SettingsScreen
```

For **web** (route hierarchy):
```
/                              → HomeScreen (public)
/login                         → LoginScreen (public)
/admin                         → AdminDashboard (requireRole=ADMIN)
├── /admin/users               → AdminUserList
│   └── /admin/users/:id       → AdminUserDetail
└── /admin/orders              → AdminOrderList
```

Authentication and role gates ARE marked on the tree (`(public)`, `(requireAuth)`, `(requireRole=…)`) so the auth surface is reviewable at a glance.

### 2.2 Screen Registry (per-screen entries)

Each screen gets an entry. Fields:

| Field | Source | Notes |
|-------|--------|-------|
| `Screen ID` | hand-curated | `SCREEN-XXX` numbering, or stable kebab-case slug (`product-detail`) |
| `Name` | hand-curated | Display name as designers / PM use it |
| `Type` | hand-curated | `Web` / `Mobile` / `Both` |
| `Path` | hand-curated | URL path for web, navigation route name for mobile |
| `Auth` | hand-curated | `Public` / `Authenticated` / `Role: ADMIN` (matches §2.1 tree gates) |
| `Description` | hand-curated | 1–2 sentences — what the user does on this screen |
| `Stories` | hand-curated | Story IDs that implement / modify this screen (back-links to backlog) |
| `API endpoints` | **DERIVED from stories** | Generated by `/screen-map` from each linked story's API Endpoints Used table — never hand-edit this column |
| `Status` | derived | Aggregated from story statuses (Todo / In Progress / Completed / Blocked) |

## 3. Source-of-Truth Split (prevents drift)

| What | Source of truth | Who maintains |
|------|-----------------|---------------|
| Navigation hierarchy (§2.1) | `screen-map.md` itself | Hand-curated by team |
| Screen metadata (Name, Type, Path, Auth, Description) | `screen-map.md` itself | Hand-curated by team |
| Story → Screen link | The story file (per `.claude/rules/screen-driven-backlog.md` `**Screen:**` field) | Updated by `/add-scope`, `/process-client-docs`, manual story edits |
| API endpoints consumed by a screen | The story file (`**API Endpoints Used:**` table) | Updated when the story is added/edited |
| API endpoints column in `screen-map.md` | **Generated** by `/screen-map` reading stories | Never hand-edited |
| Screen status | **Generated** by `/screen-map` aggregating story statuses | Never hand-edited |

**Rule:** if you need to change which API a screen calls, edit the story — `/screen-map` will pick it up on next run. If you hand-edit the API column in `screen-map.md`, the next `/screen-map` run will overwrite your edit. The tool is the single derivation path.

## 4. Lifecycle

### 4.1 Adding a New Screen

1. Add the screen entry to `screen-map.md` §2.2 (Name, Type, Path, Auth, Description) and place it in the navigation hierarchy in §2.1.
2. Add the implementing story via `/add-scope` (or hand-create per `.claude/rules/screen-driven-backlog.md`) — the story's `**Screen:**` field must match the Screen ID / name in the map.
3. Run `/screen-map` to populate the API endpoints column for the new screen from the story.

### 4.2 Modifying an Existing Screen

1. Update the story (acceptance criteria, API Endpoints table) per `.claude/rules/screen-driven-backlog.md`.
2. Run `/screen-map` — API columns and Status refresh automatically.

### 4.3 Removing a Screen

1. Mark the story as `Status: Removed` (or delete it).
2. Move the screen entry to a `## Removed Screens` section at the bottom of `screen-map.md` (do not delete — preserves history for change discussions).
3. Update the navigation hierarchy in §2.1 to remove the node.

---

## 5. Drift Detection

`/screen-map` checks for and reports four drift conditions:

1. **Story without a matching screen entry** — story has `**Screen:** X` but no entry `X` exists in the map → suggest adding the entry.
2. **Screen entry without any backing story** — entry exists but no story references it → flag as orphan (might be a planned screen, dead design, or stale entry).
3. **Story's API table differs from a previous run** — informational; expected when stories are edited.
4. **Navigation hierarchy node missing a registry entry** (or vice versa) — tree and registry must agree.

The report at the end of `/screen-map` lists all drift items so they can be resolved before the next backlog/sprint review.

---

## 6. Relation to Other Rules

- `.claude/rules/screen-driven-backlog.md` — defines the story-level structure that this rule consumes. The map is a *view* over those stories, not a replacement.
- `.claude/rules/api-first.md` — Phase A contract verification still applies per story before frontend implementation. The map does not skip this gate; it complements it.
- `.claude/rules/api-documentation.md` — endpoint docs (referenced in the API columns) remain the source of truth for what each endpoint does.
- `.claude/rules/api-versioning.md` — when an endpoint's path changes (`v1` → `v2`), stories update, and `/screen-map` reflects it on next run.
- `.claude/rules/documentation.md` §2.1 — file-size limits apply (split per-domain if a single map exceeds 300 lines).

---

## 7. When NOT to Use This Artifact

- API-only backends (no UI surface)
- CLI tools, libraries, infrastructure projects
- Simple SPAs with ≤ 5 screens — the backlog alone is enough; the map's overhead exceeds its value
- Marketing landing pages (low complexity, low API surface)

If a project starts small and grows, add the artifact later: run `/screen-map` once it's worth maintaining.
